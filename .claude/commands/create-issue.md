# 이슈 생성 및 브랜치 생성

작업을 시작하기 전에 이슈를 생성하고 브랜치를 만듭니다.

**중요**: 이 명령어는 이슈 생성과 브랜치 생성까지만 수행합니다. 작업 완료 후 `/commit-push` 명령어를 사용하세요.

## 📌 컨벤션 참고
- 브랜치: `docs/conventions.md` (브랜치 컨벤션)
- 이슈 템플릿: `.github/ISSUE_TEMPLATE/`
  - `feature_request.md` - 기능 구현
  - `refactor-issue.md` - 리팩토링
  - `docs-issue.md` - 문서 작업

---

## 🔄 작업 순서

### Step 1: 작업 내용 질문

사용자에게 질문:
- "어떤 작업을 진행하시나요?"

사용자의 답변을 바탕으로:
- 작업 타입 파악 (feature/fix/refactor/docs/chore 등)
- 작업 내용 파악

### Step 2: 이슈 타입 확인

사용자에게 **이슈 타입 확인**:
- "이슈 타입: feature/refactor/docs 중 어떤 것인가요?"

### Step 3: 이슈 내용 작성

해당 템플릿에 맞춰 이슈 내용 작성:

#### Feature (기능 구현)
```markdown
## ✨ 제안 개요
[간단한 기능 설명]

## 🛠 작업 내용
- [ ] [작업 1]
- [ ] [작업 2]

## 📌 참고 자료 (선택)
[관련 자료가 있다면]
```

#### Refactor (리팩토링)
```markdown
## ♻️ 리팩토링 대상
[대상 코드/컴포넌트]

## 📌 개선 이유
[리팩토링이 필요한 이유]

## 🛠 기대 결과
[리팩토링 후 기대 효과]
```

#### Docs (문서)
```markdown
## 📄 문서 작업 개요
[문서 작업 설명]

## 📌 범위
- [ ] README
- [ ] Wiki
- [ ] 주석

## 작업 내용 요약
[정리할 내용]
```

**사용자에게 이슈 내용 보여주고 확인 받기**:
- "다음 내용으로 GitHub 이슈를 생성하시겠습니까?"
- 이슈 내용 전체 표시

### Step 4: GitHub 이슈 생성

`gh` CLI 사용하여 이슈 생성:
```bash
gh issue create --title "타입: 제목" --body "$(cat <<'EOF'
이슈 내용
EOF
)" --label "라벨"
```

**라벨 매핑**:
- feature → FEAT
- refactor → REFECTOR
- docs → DOCS
- fix → FIX
- chore → CHORE

이슈 번호 저장 (예: #123)

### Step 5: 브랜치 생성

**중요**: 모든 새 브랜치는 반드시 **develop 브랜치에서** 생성합니다.

**브랜치 컨벤션**: `종류/이슈번호--브랜치이름` (전부 소문자)

사용자에게 브랜치 이름 확인:
- "브랜치 이름: `타입/이슈번호--제안된이름`"
- 제안된 이름은 작업 내용 기반으로 자동 생성
- 예: `feature/123--login-ui`

develop 브랜치로 이동 및 최신화 후 브랜치 생성:
```bash
git checkout develop
git pull origin develop
git checkout -b 브랜치명
```

### Step 6: 완료 안내

브랜치 생성이 완료되면 사용자에게 다음 안내 메시지 표시:

```
✅ 이슈 및 브랜치 생성 완료!

이슈: #[이슈번호]
브랜치: [브랜치명]

📌 다음 단계:
1. 작업을 진행하세요
2. 작업 완료 후 `/commit-push` 명령어를 사용하세요
```

---

## ⚠️ 주의사항

### 브랜치 네이밍
- 반드시 **소문자**
- 형식: `종류/이슈번호--이름`
- 예: `feature/24--modal-ui`, `fix/45--button-click-bug`

### 금지 사항
- main/master에 직접 브랜치 생성 금지
- develop에서만 새 브랜치 생성

---

## 📊 전체 워크플로우 요약

### 🤖 자동화 범위 (이 명령어가 수행)
1. **작업 내용 질문**
2. **이슈 타입 확인** (feature/refactor/docs)
3. **이슈 내용 작성** (템플릿 기반)
4. **GitHub 이슈 생성** (gh issue create)
5. **브랜치 생성** (develop에서 타입/번호--이름)
6. **완료 안내**

### 👤 사용자가 직접 수행
7. **코딩 작업 진행**
8. **`/commit-push` 명령어 실행**

## 💡 사용자 인터랙션

각 단계마다 사용자 확인:
1. "어떤 작업을 진행하시나요?"
2. "이슈 타입: feature/refactor/docs 중 어떤 것인가요?"
3. "다음 내용으로 GitHub 이슈를 생성하시겠습니까?" + 이슈 내용 표시
4. "브랜치 이름: `제안된브랜치명` - 맞습니까?"
5. (완료 후) "작업 완료 후 `/commit-push` 명령어를 사용하세요" 안내 표시
